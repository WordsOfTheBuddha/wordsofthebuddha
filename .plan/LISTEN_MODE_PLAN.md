# Plan: Voice Playlist & Listen Mode

A new dedicated "Listen mode" surface (`/listen/[discourse]`) that delivers hands-free, queue-driven audio playback (A3), built on the existing voice manifest + audio infrastructure. Reading mode stays as-is. Cross-discourse queue advancement happens *inside* listen mode via `history.replaceState` — no full page reloads. Persistent player shell across reading↔listening (A2) is deferred to a later phase, contingent on validated need.

Scope of v1: nav-mode queue (derived from `routes.ts`) + anthology queues. User-curated playlists (C), MP3 export, and podcast feeds are explicitly out of scope.

### URL scheme (confirmed: keep current)

- Review-room: **`/review-room?pli=true#{id}`** where `{id}` ∈ {`save`, `readlater`, `<userPlaylistId>`, plus existing built-in views like `read`}.
- The `?pli=true` flag is a user preference (panel-layout-inset). It rides through unchanged; we read the hash for the active id and ignore the flag in routing logic.
- Reserve hash ids `save`, `readlater`, `read` as built-ins; user playlist ids must not collide. Generator slug-cases names and appends a numeric suffix on collision.
- Listen-mode reuses `?pl={id}` with the same id space.
- **No URL migration** in this phase — keep current `/review-room?pli=true#…` family.

### Data model (Firestore)

```
users/{uid}/playlists/{playlistId}
  {
    id: string,                  // mirrors doc id; stable, slug-cased
    name: string,                // display name (≤ 200 chars)
    builtin: "save" | "readlater" | null,  // non-null = pinned, undeletable
    sortMode: "added" | "updated" | "manual",  // default "added"; only "manual" persists effective ordering
    lastListenedSlug: string | null,  // per-playlist resume
    createdAt: Timestamp,
    updatedAt: Timestamp,
    itemCount: number,           // denormalised for list views
    coverSlug: string | null,    // first item, optional
  }

users/{uid}/playlists/{playlistId}/items/{slug}
  {
    slug: string,                // discourse slug; doc id = slug for O(1) membership check
    addedAt: Timestamp,
    order: number,               // sparse float (1024-step gap insert) — see "Ordering"
  }

users/{uid}/lastListened
  {
    slug: string,                // global last-listened (independent of any playlist)
    at: Timestamp,
  }
```

Two built-ins are seeded lazily on first write (idempotent upsert with `builtin: "save" | "readlater"`).

**Why a subcollection over an array on the parent doc?**
- Items grow unboundedly; arrays would balloon the parent doc and cost re-reads on every change.
- Per-item docs let us key by `slug` for O(1) "is this discourse in this playlist?" queries (used for the dropdown checkbox state).
- Reordering writes only the moved item's `order`, not the whole list.

### Ordering — sparse float (Option B)

Each item has `order: number` initialised to `(prevOrder + nextOrder) / 2`; new appends use `lastOrder + 1024`. When the gap between two adjacent items drops below `1e-6`, run a one-shot rebalance (`order = i * 1024`) over the whole playlist. This is cheap (≤ 500 items per quota) and rare.

The client orders by `order` ascending **only when `sortMode === "manual"`**. For `added` / `updated`, sort client-side by the timestamp field — `order` is ignored but kept untouched, so switching back to manual restores the user's last manual arrangement.

### Quotas (confirmed)

- **10 playlists per user** (excluding the two built-ins).
- **500 items per playlist**.
- **200-char playlist name**.
- All enforced at write time in the service layer with a friendly toast on hit.

### Sync layer

`src/service/playlists.ts` exposes:
```ts
listPlaylists(): Promise<UserPlaylist[]>             // /users/{uid}/playlists, builtin first then by name
getPlaylist(id): Promise<UserPlaylistWithItems>      // includes items in current sort order
addItem(id, slug): Promise<void>                     // upserts items/{slug}, bumps itemCount
removeItem(id, slug): Promise<void>
reorder(id, slug, newOrder): Promise<void>           // single-doc write
setSortMode(id, mode): Promise<void>
setLastListened(id, slug): Promise<void>             // per-playlist resume
setGlobalLastListened(slug): Promise<void>           // independent of playlist
createPlaylist(name): Promise<string>                // returns id; quota-checked
renamePlaylist(id, name): Promise<void>              // user playlists only (builtin rename deferred)
deletePlaylist(id): Promise<void>                    // forbidden for builtins
```

All writes go through `runTransaction` (or batched write) so `itemCount` and the items subcollection stay consistent. Cache active user's playlists in `IndexedDB` (existing offline scaffolding) for instant boot and offline read; mutations queue and replay on reconnect.

### Listen-mode integration

`?pl=<id>` resolution layer:
```ts
async function resolvePlaylist(id: string): Promise<Playlist | null> {
  if (id === "save" || id === "readlater") return await fetchUserPlaylist(id);  // built-in
  const anthology = getStaticPlaylist(id);
  if (anthology) return anthology;
  return await fetchUserPlaylist(id);  // user playlist by id
}
```

- Anthology playlists remain build-time bundled (zero-fetch boot, public).
- Built-in + user playlists are auth-gated and fetched on listen-mode boot.
- Drawer header shows the playlist name (already wired via `#listen-queue-title`).
- Playback transitions preserve `?pl=` (already wired).
- Drawer reorder + remove enabled for user playlists + built-ins; disabled for anthologies.

### Permissions / security

Firestore rules:
```
match /users/{uid}/playlists/{pid} {
  allow read, write: if request.auth.uid == uid;
}
match /users/{uid}/playlists/{pid}/items/{slug} {
  allow read, write: if request.auth.uid == uid;
}
match /users/{uid}/lastListened {
  allow read, write: if request.auth.uid == uid;
}
```
No public reads. Sharing is an explicit future feature (Phase 6.x — share a playlist by writing a denormalised public `shared_playlists/{id}` doc on demand).

### Implementation order

1. **Firestore schema + service module** (`src/service/playlists.ts`) with offline cache. Seed built-ins lazily; back-fill from any existing legacy `saves` / `read-later` arrays on first sign-in post-deploy.
2. **Save dropdown** on the reading-page toolbar (replaces today's Save button; `Read Later` standalone button stays). Smallest user-visible surface; reuses existing toggle plumbing for the two built-ins.
3. **Review-room rendering for user/built-in playlists** at `/review-room?pli=true#{id}` — existing PostCard grid + new toolbar (Play all / Shuffle / Sort) + Continue-from chip.
4. **Manual sort + drag-reorder** in review-room and listen-mode drawer (shared `reorder` write path). Sparse-float ordering with rebalance.
5. **Rename + Delete** in review-room (user playlists only).
6. **`+ New playlist`** inline-create in Save dropdown.
7. **Listen-mode user-playlist support** — `resolvePlaylist` extension + drawer reorder/remove + per-playlist resume on boot.
8. **Quota enforcement + UX toasts**.

### Out of scope for v1

- Built-in rename (revisit later; built-in semantics referenced by id, name change is cosmetic).
- "Add whole collection to playlist" bulk action.
- URL migration to a cleaner scheme.
- Cross-user sharing.
- Read state and Highlights as playlists (they remain derived views).

## Decisions

- **Listen mode is a separate surface (A3), not just a UX layer on reading pages.** Reading and listening are distinct modes.
- **URL: path-based `/listen/[discourse]`, query for context (`?pl=...&i=...`).** Cacheable, prerenderable, shareable.
- **No subdomain.** Cost > benefit for shared data + SW + auth.
- **Autoplay always-on** (toggle UI removed). The queue chip opens the drawer; `Browse` label when in nav-mode with no successor.
- **Queue chip opens drawer** (Phase 2.4 landed).
- **Queue prev shows all routes-prev items, dimming unplayed.** Symmetric and predictable over "only-played" history.
- **No drawer reordering in v1.** Meaningless in nav/anthology mode; defer until user-playlists exist.
- **WebM/Opus stays primary.** No MP3 export in v1.
- **A2 (persistent shell) cancelled.** No parallel reading-page audio surface to persist.
- **User playlists (C) deferred** to post-v1.

## Further Considerations

1. **End-of-playlist behavior in anthology mode.** Stop with toast / loop / fall through to `routes.ts`? Recommend: **stop with a one-tap "Continue with related"** that switches to nav-mode queue. Prevents surprise.
2. **Pre-roll generation cost.** Generating one intro audio per discourse via TTS adds ~1000 small files and re-runs of `generate_voice.py`. Recommend: **defer to Phase 4** — pre-rolls add the most value in queues, less so for nav-mode where adjacent discourses are obviously sequential.
3. **Listen-mode artwork.** Use existing book/anthology cover (e.g., `noble-truths-noble-path-book-cover.jpeg`) as MediaSession artwork? Recommend: **per-discourse artwork if available, fallback to a single project-level image.** Avoids per-track artwork generation.
